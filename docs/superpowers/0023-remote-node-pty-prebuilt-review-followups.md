## FINDING-003 — MEDIUM / quality
- **Location:** -
- **Claim:** coProvisionNodePty (bootstrap.ts) narrows decideNodePtyProvision's 4-variant ProvisionDecision result to just 'skip' vs (implicitly) 'install' with a comment ('selection.ok narrows decideNodePtyProvision to skip|install only') instead of an exhaustive switch or a defensive assert. TypeScript does not actually enforce this cross-function invariant -- decideNodePtyProvision's return type still includes 'proceed-unmanaged'/'no-match'. If that pure function is ever edited (e.g. a future arm64/musl fast-follow changes the decision table) so it returns 'proceed-unmanaged' or 'no-match' while selection.ok is true, this call site would silently fall through into walkBundleDir/encodeNodePtyPayload/runNodePtyInstall and attempt an install nobody asked for, rather than failing loudly -- no test exercises this fallthrough because the current pure implementation never produces it.
- **Proposed fix:** Replace the comment-only narrowing with an exhaustive switch/if-chain over decision.kind (skip -> return undefined; install -> proceed; default -> throw an internal-invariant error naming the unexpected kind), so a future change to decideNodePtyProvision that violates the selection.ok invariant fails fast with a clear diagnostic instead of silently attempting an unintended install.

## FINDING-006 — MEDIUM / security
- **Location:** -
- **Claim:** release.yml's new build-node-pty-linux job installs Node inside the ubuntu:20.04 build container via 'curl -fsSL https://deb.nodesource.com/setup_22.x | bash -' -- a curl-pipe-to-bash of a third-party script, fetched with no signature or checksum pinning, executed with root privileges in the same container that subsequently compiles the node-pty native binary shipped inside every released installer and pushed onto every remote host by this feature. A compromise of deb.nodesource.com, its CDN, or the network path (DNS hijack, TLS-terminating proxy on the runner) yields arbitrary code execution in the trusted release pipeline that produces a binary this feature auto-uploads to customer remote hosts with no further human review.
- **Proposed fix:** Replace the curl|bash Node install with a pinned, checksum-verified artifact: use actions/setup-node (which verifies against Node's published checksums) or download the NodeSource .deb/tarball explicitly and verify its sha256 against a value recorded in the workflow before installing, rather than piping a fetched script directly into bash.

## FINDING-007 — LOW / security
- **Location:** -
- **Claim:** release.yml's node-pty build/stage steps fetch the pinned node-pty version via 'npm pack node-pty@$VERSION' (twice: once in build-node-pty-linux, again in the release job's staging step) resolved only by version string, never checked against the sha512 'integrity' hash package-lock.json already records for that exact node-pty version (verified present at package-lock.json:5225). A registry-side substitution for that version, or a compromised registry mirror/proxy reachable from the GitHub-hosted runner, would let a tampered node-pty source ship as the release's native prebuilt without detection, since nothing in the release lane ties the fetched tarball back to the project's own vetted lockfile hash.
- **Proposed fix:** After 'npm pack', compute the tarball's sha512 and compare it to the 'integrity' field recorded for node-pty in package-lock.json (parsed via node -p or a small script) before building from it; fail the job on mismatch.

## FINDING-008 — LOW / determinism
- **Location:** src/remote-client/bootstrap.ts:355-366 (walkBundleDir); scripts/stage-node-pty-prebuild.mjs:64-72 (copyTree)
- **Claim:** walkBundleDir (bootstrap.ts) and copyTree (stage-node-pty-prebuild.mjs) enumerate directory entries with UNSORTED readdirSync. In walkBundleDir this order fixes BOTH the NODE_PTY_PAYLOAD_V1 header entries AND the concatenated file-byte order of the upload payload, so the exact bytes streamed over the ssh install channel vary by filesystem/platform/run (readdir order is unspecified and OS-dependent). It is functionally harmless TODAY (header and bytes derive from the same walk so they stay self-consistent; UNPACK_SRC restores files by header path; the only integrity anchor is pty.node's sha-256; and every frozen test sorts its tree comparison, e.g. node-pty-probe-unpack.test.ts's tree() and the .sort()'d expected lists) — but it is a latent nondeterminism hazard: a future whole-payload hash, a reproducible-build assertion, an upload-dedup/cache keyed on payload bytes, or any test that pins the wire bytes would break intermittently. Same nondeterministic-ordering class in copyTree's staged-bundle copy order.
- **Proposed fix:** Sort the readdir result before recursing/emitting in both spots, e.g. 'for (const name of readdirSync(d).sort())', so the streamed payload and the staged bundle are byte-order-reproducible across hosts and runs. propose CONV: when a filesystem directory listing (readdir/readdirSync) feeds a wire payload, a hash input, or a reproducible build artifact, sort the entries by name before use — never rely on the platform/filesystem enumeration order.

## FINDING-011 — LOW / networking
- **Location:** src/remote-client/bootstrap.ts:420-474; src/remote-client/ssh-command.ts:82
- **Claim:** The new mandatory probe gate runs before every node-pty connect with no internal timeout and no ssh liveness options (buildSshExecArgv emits no ServerAliveInterval/ServerAliveCountMax/ConnectTimeout, and there is no setTimeout anywhere in src/remote-client). A stalled remote node, or a half-open TCP connection (network dropped after auth), leaves runNodePtyProbe pending forever; the connect hangs indefinitely and only an external AbortSignal recovers. This hang surface pre-dates the feature (F19 connect/provision share it), but the probe adds a serial precondition to every connect that amplifies it. REQ-016/REQ-017 designed cancellation solely around AbortSignal, so a caller with no wired cancel has no recovery.
- **Proposed fix:** Give the probe and install channels a bounded liveness guard: either an internal timeout that settles a specific actionable fatal, or pass ssh -o ServerAliveInterval and -o ServerAliveCountMax (and/or -o ConnectTimeout) in the exec argv so a dead connection self-terminates while still permitting interactive 2FA/hardware-key auth (do not add BatchMode).

## FINDING-012 — LOW / networking
- **Location:** src/remote-client/bootstrap.ts:600-601; readLocalManifest 376-409
- **Claim:** The client uploads the actual local build/Release/pty.node bytes but stamps the NODE_PTY_PAYLOAD_V1 header with the manifest-claimed ptyNodeSha256 (encodeNodePtyPayload(files, local.manifest.ptyNodeSha256)) without ever verifying the two agree. readLocalManifest validates structure and existence only and never re-hashes the local pty.node. ## FINDING-014 — MEDIUM / devils-advocate
- **Location:** src/remote-client/bootstrap.ts:562 (coProvisionNodePty); ssh-command.ts:82 (buildSshExecArgv) / REQ-016
- **Claim:** The co-provision design opens a separate ssh exec channel (the read-only probe) before EVERY connect, including steady-state reconnects -- the REQ-016 acceptance (TEST-2344) pins the second-connect ledger as [probe, launch]. Each exec channel is an independent system-ssh invocation with no ControlMaster/multiplexing (buildSshExecArgv emits no -o ControlPath), so it re-runs authentication. For a host with 2FA / hardware-key / password auth, the probe adds one extra interactive auth challenge to EVERY reconnect forever (steady-state channels 1 -> 2; fresh-host 3 -> 5). The intake/spec acknowledge per-channel prompts inherited from F19 but neither count the new per-connect total nor mitigate the amplification the always-on probe introduces.
- **Proposed fix:** Amortize the added auth cost. Options: (a) multiplex a connects exec channels over a single ssh ControlMaster/ControlPath session so probe+install+launch+upload authenticate once; (b) fold the probe into the launch, or cache a recent skip decision keyed by host+agentDir+version, so a steady-state reconnect opens no extra channel; or (c) at minimum, state the new per-connect channel/prompt count in REQ-016 as an explicitly accepted tradeoff for interactive-auth users.

## FINDING-015 — MEDIUM / devils-advocate
- **Location:** src/remote-client/prebuilt.ts:145-149 (PROBE_SRC glibc); deriveLibc:392 / REQ-008, REQ-010
- **Claim:** PROBE_SRC derives glibc SOLELY from process.report.getReport().header.glibcVersionRuntime, defaulting to null whenever process.report is unavailable or getReport is not a function. deriveLibc then returns non-glibc, so a genuine linux-x64-GLIBC host whose node cannot self-report glibc (an older or restricted node -- precisely the locked-down/offline enterprise hosts this feature exists to serve) is classified linux/x64/non-glibc and rejected as no-match FATAL, with a diagnostic naming libc=non-glibc. At minimum, name process.report unavailability as a candidate cause in the no-match wording so a genuine glibc host is not turned away by a misleading message.

## FINDING-018 — LOW / networking
- **Location:** src/remote-client/prebuilt.ts:300-304,346-348 (UNPACK_SRC staging/promote catches); bootstrap.ts:533-537 (93 client wording) / REQ-015
- **Claim:** UNPACK_SRC folds every NON-collision remote filesystem failure during staging and promote into NODE_PTY_BYTES_EXIT (93). Add an unpacker-under-local-node test that rigs a promote/staging fs error and asserts the diagnostic is not the transfer/byte-count "retry" wording. propose CONV: a remote-side failure sentinel must map to client advice matching the failure's persistence -- never advise "retry" for a non-transient remote condition (disk full / permission / read-only).

## FINDING-019 — LOW / networking
- **Location:** src/remote-client/prebuilt.ts:306-349 (UNPACK_SRC amended promote / collision handler) / REQ-015
- **Claim:** The amended race-tolerant promote resolves a rename collision by reading the now-present finalDir marker EXACTLY ONCE (prebuilt.ts:321-329) and treating an absent/unparseable marker as a DIVERGENT install => exit 95. That marker re-read is itself an unguarded TOCTOU: between the failed rename and the readFileSync, a THIRD concurrent connect (or the winner's own next reinstall) can be mid-fs.rmSync(finalDir,{recursive}) or between its own rm and rename, so the loser reads a missing/partial marker (observedSha=null) and reports exit 95 for what is actually an IDENTICAL install -- failing a connect that REQ-015's idempotency guarantee ("two racing connects shipping the same release BOTH complete") intends to succeed. The mandated concurrency acceptance vectors (TEST-2360/2362/2363) interpose exactly ONE other promoted install and never exercise a third promoter rm-ing finalDir during the marker read, so the window passed the gate untested. Impact is bounded: the connect self-heals on the next reconnect (the probe then sees the identical install and decides skip), so the effect is a spurious one-shot connect failure under 3+ simultaneous connects to the same host+agentDir -- but exit 95's diagnostic ("another connect got there first with a divergent payload") is then simply false.
- **Proposed fix:** Make the collision resolution robust to a racing re-remove: retry the marker read a bounded number of times, and/or when the marker is missing (finalDir absent => no winner currently present) attempt one more rename of the temp into place before concluding "divergent" -- only conclude exit 95 when a stable, readable marker with a genuinely different ptyNodeSha256 is observed. Add a 3-way concurrency vector (a third promoter rm-ing finalDir during the loser's marker read) asserting identical installs never spuriously exit 95.

### FINDING-025 — MEDIUM / determinism
- **Location:** tests/node-pty-provision-flow.test.ts:400 (TEST-2351)
- **Claim:** TEST-2351 aborts the in-flight probe with a bare wall-clock timer, setTimeout(() => ctl.abort(), 300), then asserts kinds() === [probe]. This races the fake-ssh child: the probe ledger line is appended by fake-ssh.mjs at its own startup (line 140), and the same file documents this box measures the probe leg at ~250 ms (dominated by node child-spawn latency), leaving only a ~50 ms margin before the 300 ms abort. If spawn latency exceeds ~300 ms on a loaded/slow machine, the abort kills the fake-ssh child before it logs, so kinds() === [] and the toEqual([probe]) assertion fails nondeterministically. The tests phase itself flagged THIS exact 300 ms timer as one that provably races this box 250 ms probe leg and hardened the sibling TEST-2352 to synchronize on the observed ledger via waitUntil(); TEST-2351 was left on the bare timer even though the deterministic waitUntil helper already lives in the same file (line 154).
- **Proposed fix:** Mirror the TEST-2352 hardening: replace the wall-clock timer with await waitUntil(() => kinds().includes(probe), the probe channel to open) then ctl.abort(). Because the stall rig holds the probe channel open forever after it logs probe, aborting after the ledger line deterministically hits the in-flight probe and yields kinds() === [probe] on any machine. propose CONV: a test that must act mid-async-operation (abort/kill/mutate while a channel is open) MUST synchronize on an observed signal (a ledger line, a state file, or an awaited event), never a fixed wall-clock timer racing process spawn or channel setup.

### FINDING-026 — MEDIUM / devils-advocate
- **Location:** src/remote-client/prebuilt.ts:201 (PROBE_SRC emits node:process.version, never consumed); decideNodePtyProvision / REQ-008, REQ-021
- **Claim:** The load-bearing core assumption (intake decision 2, concept D2) is that one prebuilt per (os,arch,libc) is portable across ALL Node versions because node-pty is N-API (node-addon-api ^7.1.1, verified in node_modules). But N-API portability has a FLOOR: a pty.node compiled against node-addon-api 7 requires the remote node to support that N-API version (roughly Node >= 12.22 / 14.17). A remote below that floor — exactly the locked-down/offline/old enterprise hosts this feature targets — passes the probe (linux,x64,glibc all detected), gets an install, promotes and sha-verifies fine, then the agent dlopen of pty.node fails with an opaque N-API/ABI error like 'The module was compiled against a different Node.js version' / 'Node-API version X not supported'. That stderr matches NEITHER GLIBC_NOT_FOUND_RE (so glibcFloorHint stays empty) NOR MODULE_RESOLUTION_FAILURE_RE (so no recovery cycle), so resolveLaunchFatal passes the raw stderr through with zero actionable guidance and no floor hint. The glibc-too-old floor got an entire MUST (REQ-021) with a dedicated hint; the analogous Node/N-API-too-old floor — the SAME 'host is older than the shipped binary supports' class — got nothing, even though PROBE_SRC already collects node:process.version (line 201) which is then dead data, read nowhere in the flow. The spec never states a minimum remote Node version and no test exercises an N-API-incompatible host.
- **Proposed fix:** Add a Node/N-API floor analogous to REQ-021's glibc floor: either (a) have decideNodePtyProvision/classify compare the probe's node:process.version against the minimum Node the shipped prebuilt requires and no-match FATAL early with an actionable diagnostic (naming the detected node version, the required minimum, and the escape hatch), or (b) detect the N-API/ABI-class dlopen stderr in resolveLaunchFatal and append a floor hint like the glibc one, and never enter the recovery cycle for it (reinstalling the same binary cannot help an old runtime). Document the minimum supported remote Node version in the spec/feature doc. propose CONV: when shipping a prebuilt native binary a remote will dlopen, the client MUST diagnose every too-old-runtime floor it can hit (glibc AND Node/N-API ABI) with a specific actionable message, never pass an opaque loader error straight through.
