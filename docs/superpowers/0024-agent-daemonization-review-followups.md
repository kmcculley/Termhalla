## FINDING-006 — MEDIUM / determinism
- **Location:** src/agent/daemon-server.ts:315-340 (claimSocket) + src/agent/bridge.ts:145-164
- **Claim:** Stale-socket RECLAIM removes agent.sock/daemon.json OUTSIDE any serialized critical section. In claimSocket the sequence [last probe returns not-connectable] -> readMetadataSafe -> decideDaemonReach -> safeUnlink(socketPath)/safeUnlink(metadataPath) is unguarded across processes: the probe loop is a heuristic mitigation, not a mutex. Under adversarial interleaving a concurrent winner W that binds in the window (or between W's listen() and W's later writeFileSync(metadata) where daemon.json is briefly ABSENT) is seen by the loser as 'refused + absent metadata' -> reclaim -> unlink of W's LIVE socket, after which the loser rebinds the freed path => TWO listeners bound to the same agent dir (split-brain), and later W.cleanup()/idle-exit unconditionally unlinks the path it may no longer own, orphaning a daemon (the exact orphan-accumulation locked decision D3 rejects). This violates REQ-005 'at most ONE daemon' under concurrency and CONV-045 (a client-side pre-write check with no server compare-and-set MUST be inside the write's serialized critical section OR the residual race MUST be disclosed - neither holds; the spec asserts race-proofness). REQ-007's acceptance requires the reclaim-vs-concurrent interleaving to 'ride the REQ-005 vector', but the only interleaved real-process vector TEST-2424 starts two daemons on a FRESH dir (no remnants -> reclaim branch never taken) and TEST-2425 exercises reclaim only SEQUENTIALLY (single bridge, pre-seeded remnants) - so no test enforces the 'provably impossible by the REQ-005 guarantee' clause the spec relies on (CONV-060).
- **Proposed fix:** Serialize reclaim+rebind behind a real critical section keyed on the agent dir - the plan's Q3-anticipated O_EXCL lockfile whose holder alone may unlink stale remnants and then bind, so a stale-remnant unlink can NEVER remove a concurrent winner's live socket; and make cleanup()/idle-exit only unlink files this process still owns (re-check ownership, e.g. metadata pid == process.pid, before removal). If a residual race is instead accepted, disclose it explicitly per CONV-045. Add an interleaved reclaim-under-concurrency vector per REQ-007+CONV-060: pre-seed dead-pid remnants AND start two real --daemon/--attach processes with overlapping bind windows, asserting exactly one listener and no live socket ever unlinked. propose CONV: a filesystem-remnant reclaim (unlink-then-recreate of a socket/lock/marker file) guarding a single-instance guarantee MUST perform the unlink inside the same serialized critical section that performs the recreate, and MUST only remove a remnant it has proven it owns - never on the strength of a prior liveness probe alone.

## FINDING-020 — MEDIUM / enterprise-arch
- **Location:** src/agent/daemon-server.ts:115-142 (createDaemonLogSink.append)
- **Claim:** createDaemonLogSink.append rewrites the ENTIRE in-memory buffer (up to DAEMON_LOG_MAX_BYTES = 1 MiB) to disk on EVERY call via writeFileSync(logPath, buf). Once the ring buffer fills, each single diagnostic line forces a full ~1 MiB file rewrite. session.ts emits a diag per skippable-malformed frame (frame skipped, session continues at session.ts:259) WITHOUT ending the connection, and daemon-server emits a diag per failed send, so a peer streaming malformed-but-skippable frames or a flapping/reconnecting client drives per-frame ~1 MiB rewrites — an owner-local write-amplification / disk-thrash vector on the remote host that degrades the very long-lived survival daemon this feature introduces (a thrashing daemon serves its live sessions slowly). The FINDING-010 fix bounded log GROWTH but its full-rewrite-per-line implementation makes each diagnostic cost O(cap) disk IO. TEST-2445 asserts the file-size cap but never the write VOLUME, so the amplification is uncaught.
- **Proposed fix:** Append incrementally (appendFileSync or an append-mode fd) and only rotate (rename to a single .prev, start fresh) when the file crosses the cap, instead of a whole-buffer writeFileSync per line — turning O(cap)-per-append into O(line)-per-append with an O(cap) cost only at each rotation. Add a test that bounds total bytes written for N appends. propose CONV: a size-capped diagnostics/ring log for a long-lived process MUST append incrementally and rotate at the cap, never rewrite the whole capped buffer to disk on every line (write amplification scales per-line IO to the cap size).

## FINDING-021 — LOW / enterprise-arch
- **Location:** src/agent/daemon-server.ts:115-142,441-475 (createDaemonLogSink / runDaemon) + REQ-004
- **Claim:** The headless survival daemon is under-instrumented and self-erasing for a process meant to run unattended for days on a remote host. (a) createDaemonLogSink truncates daemon-<ws>.log at each daemon start (REQ-004 MUST), so a crashed daemon crash output (routed to the log per the FINDING-011 fix) is WIPED by the very next connect, which spawns a fresh daemon whose log-sink constructor does writeFileSync(logPath, '') — i.e. the forensic trace is destroyed exactly when an operator reconnects to investigate the lost session. Now that the size cap (FINDING-010) independently bounds growth, truncate-at-start no longer serves growth-bounding; its only realized effect is to defeat FINDING-011 forensic intent in the crash-then-reconnect path (the common case). (b) The daemon writes NO positive lifecycle lines (start with pid/version/proto, socket claim/bind, idle-exit reason, lease-steal, connection establish/end) — only error diagnostics — so on the healthy path daemon-<ws>.log is EMPTY and cannot confirm the daemon ever ran or why it exited, weakening operability of an invisible long-lived process.
- **Proposed fix:** Preserve the prior generation on start (rename daemon-<ws>.log to daemon-<ws>.log.prev before truncating) so a crash trace survives the reconnect that spawns the successor; and emit positive lifecycle diagnostics (start pid/version/proto, claim/bind, idle-exit reason, lease-steal, connection-count transitions) so the log is a usable operability record on the healthy path too. propose CONV: a headless long-lived daemon/service MUST preserve the previous generation diagnostics across a restart (never truncate the only crash trace in place) and MUST log its own lifecycle transitions (start/ready/exit-reason), not only error diagnostics.

Add a manager-routing assertion that after a forget disconnect with tracked panes, owns() returns false for those pane ids.

### FINDING-030 — LOW / enterprise-arch
- **Location:** src/agent/daemon-lifecycle.ts:83-85 (maybeArm) + REQ-019 + src/renderer/store.ts closeWorkspace
- **Claim:** REQ-019 states an unreopened workspace daemon is idle-reaped once its established-connection count reaches 0, but daemon-lifecycle maybeArm arms the idle timer only when establishedCount===0 AND paneCount===0. A close-tab detach ends the connection (establishedCount -> 0) yet leaves the panes running on the daemon (paneCount > 0), which is precisely the survival the feature exists to provide, so the daemon does NOT idle-reap for the flagship case (a detached workspace with a running claude or an interactive shell). The spec reap claim therefore overstates automatic reclamation: for the common detach-with-live-panes case the daemon persists until the pane process exits or a manual reap. Compounding this, there is no app-side affordance to enumerate or reap detached remote daemons/sessions per host, no server-side max-detached-age or max-session cap, and no user feedback on close that a remote session was left running. Over time a user accumulates invisible, uncapped, long-lived remote daemons + PTYs across closed-but-never-reopened workspaces, reclaimable only by the pane own exit or a manual SSH kill, a material operability/scalability change from pre-0024 clean death-on-disconnect for this default gesture. Distinct from the deferred FINDING-021 (log diagnostics): this is fleet visibility + reap + an inaccurate spec reap claim, not diagnostics.
- **Proposed fix:** Correct the REQ-019 reap wording to state reap requires zero established connections AND zero live panes (so a detach with running panes persists by design), disclose in remote-workspaces.md per CONV-054 that a close-tab detach leaves the remote daemon + PTYs running until reopened, the pane exits, or a manual reap, and track a follow-up for app-side detached-session enumeration/reap and/or a server-side max-detached-age cap so the fleet stays bounded and operable. propose CONV: a survival/persistence feature that leaves server-side processes running after the client default going-away gesture MUST provide, or explicitly defer with disclosure, a way to ENUMERATE and REAP those processes from the client - a persistent process the user cannot see or stop from the app is an operability orphan.

### FINDING-031 — MEDIUM / networking
- **Location:** src/agent/daemon-server.ts:334-351,541-563 (listenOnce / runDaemon server wiring)
- **Claim:** The long-lived daemon's net.Server has NO 'error' handler after listen() succeeds, and main.ts installs no process-level uncaughtException handler. listenOnce attaches server.once('error', onError) only for the bind, then server.removeListener('error', onError) in the listen callback (line 347); runDaemon thereafter only wires server.on('connection') (line 541) and per-socket sock.on('error') (line 562) — the SERVER object itself is left with no 'error' listener for its entire serving lifetime. A post-listen server 'error' event (Node emits one on an accept-time failure such as EMFILE/ENFILE under file-descriptor exhaustion — plausible for a survival daemon that runs for days accumulating connections/panes/FDs, or other libuv accept errors) is therefore an unhandled EventEmitter 'error' → thrown → uncaught exception → the whole daemon process dies. That kills EVERY surviving PTY for that workspace (a running claude included), which is precisely the catastrophe daemonization exists to prevent. It also introduces a third daemon-exit path that REQ-002's invariant forbids ('the daemon exits ONLY via REQ-006 idle/signal — never because a connection ended'): an accept-error crash is neither idle nor signal.
- **Proposed fix:** Re-attach a durable, process-surviving server-level error handler immediately after claimSocket returns the bound server in runDaemon, e.g. server.on('error', (e) => appendLog('daemon server error (accept/listener): ' + String(e))) so an accept-time error is logged and the daemon keeps serving its existing panes/connections instead of crashing. Optionally add a top-level process.on('uncaughtException'/'unhandledRejection') backstop in main.ts's daemon path that logs to the daemon log and keeps the process alive (panes survive) rather than tearing down. Cover with a unit that emits a synthetic 'error' on the server after listen and asserts the process/store survives. propose CONV: any long-lived net.Server/listener (a process meant to outlive individual connections) MUST retain a process-surviving 'error' handler for the whole serving lifetime — never leave the listener with its bind-time error handler removed and none re-attached, so an accept-time error (EMFILE/etc.) cannot crash the process.
